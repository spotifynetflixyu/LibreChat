【search_price_candidates 通用查價規則】

一、批次與查詢

1. 同一輪材料放入一次 `queries`，加工放入同一 tool call 的 `processingQueries`；兩者皆依各自 `queryResults` 順序對回，不跨 query 混用候選。
2. 已知類別首查必須帶 `category`，再依類別規則加入可確認的 structured filters。只有類別未知時才先用 `mode: "category_discovery"` + 通用 `keyword`。
3. `erpItemCode` 只供後續回合精確重取已確認候選；首查不得使用，本輪已返回的候選不得重查。
4. `subcategory` 不確定時省略。keyword 只放 category／subcategory 尚未表達的通用產品特徵或規格；不得自行加入專有品名、廠牌、系列、來源 DB token、ERP／表單代號。只有使用者原文明示時可保留專有詞；候選 product_name 不得反向成為重查 keyword。ST50、SN400B 只留在備註。
5. 平時省略 `limit`；只有前次結果已達30且仍無精確可報價候選時，才提高至100。
6. 已有規格精確且可報價候選就直接選價。只有 `no_match`、無精確候選、精確候選全不可報價或缺必要規格時可修正一次；不得原樣重送、夾帶成功 query 或只改 limit。仍失敗即人工複核。

二、選價與計算

1. 採符合客戶等級的 `tierPrices` 與候選有效 `unit`；忽略 raw ratio 欄位。缺價為未知，不是0；候選標示不可報價時不得使用。
2. 未指定厚度時，在其他條件相符且可報價候選中採最小厚度；孔加工區間與細項規則除外。無法唯一確認即人工複核。
3. Kg／M 分別以總 kg／總 m 乘單價；其他單位原則為整數數量 × direct 單價。重量、長度、單價保留原精度，每一材料列最終小計各自 `ceil` 成 TWD 整數；加工列分開進位。
4. 材質、規格、unit、配料、計重或加工有類別規則時，以該細項規則為準；通用規則不得覆蓋其明示例外。
5. C型鋼與H型鋼以 Kg 價為主；無可用 Kg 價即人工複核，不自動改用 M、支或其他 unit。只有訂單明示「不切清」時，才採相符整件 direct 單價。

三、切工

材料切工只使用同次結果的 `cuttingPrices`；已返回者不得另查，未返回者不得自行借用相近切工。刀數、加價與限制依【長條料切工規則】。

四、加工價格

1. `processingPrice` 只使用同次返回的通用加工與指定鋼材可用加工；AI 不得把非鋼材 subcategory 一律視為通用。
2. 一次可帶最多3筆 `processingQueries`。每筆 `categories` 可同時列多個訂單鋼材類別，`processingCategories` 可同時列切工、孔、倒角、開槽、折工、焊接或其他；`keyword` 只使用通用加工詞。`processing_method`、`processing_shape` 已在 normalized text，不新增 query params。
3. 首查不得使用頂層 `productNames`。加工探索不超過10筆會直接返回價格；超過10筆只返回全部唯一 `productNames`。清單中的每個值都是 `steel.prices.product_name` 完整原值。只有收到這份清單後，AI 選定品名並 requery：`{"productNames":["品名A","品名B"]}`，以 exact equality 取得所有指定材料或加工品名價格，不作 keyword、alias、模糊或 normalized text 比對。此 requery 不帶 `queries`、`processingQueries` 或其他 params；收到 productNames 不是 no_match 或 error。
4. 同一加工若也出現在 legacy `cuttingPrices`，只採一個來源，不得重複計價；v4.4 `processingPrice` 有精確 ERP row 時優先使用。
