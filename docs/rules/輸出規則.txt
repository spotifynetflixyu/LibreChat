【Steel 輸出規則】

一、輸出表單範圍
1. AI 的報價輸出表單只處理四張 active sheets：`system_order`、`customer_data`、`manual_review`、`customer_quote`。
2. AI 只需要輸出本輪要更新的 sheets；整張 active sheet 未輸出時，backend 沿用上一版 active sheet。
3. 只要 AI 輸出某張 sheet，該 sheet 就是完整覆蓋版，不是逐列 merge。上一版同 sheet 有、但本輪輸出 sheet 未列出的列，代表該列已清空或刪除。

二、客戶等級與報價同步
1. 若本輪更新 `customer_quote`，且客戶價格等級有改變或會影響報價價格，必須同步輸出更新後的 `system_order` 與 `customer_quote`。
2. `system_order` 的客戶 / 計價基準 / 採用價格上下文必須與 `customer_quote` 使用的客戶等級一致。
3. `customer_quote` 的單價、小計與報價總額必須依同一個已採用客戶等級重新計算。
4. 若客戶等級不確定，預設使用價格B產生 `system_order` 與 `customer_quote`；可在 `manual_review` 標示客戶或等級待確認，但不得因等級不確定而停止報價。

三、OCR 確認回覆格式
1. 本輪有 PDF / image OCR 時，第一輪回覆只輸出 OCR 結果確認表，不輸出正式報價表。
2. OCR 結果確認表應盡量保留來源粒度：來源檔案、頁碼或圖片、表格名稱、來源列號、件號 / 品項、品名或規格、材質、厚度、尺寸、數量、重量、`孔數 / 件`、`總孔數`、孔徑、孔型、切工、折工、開槽、其他加工、低信心欄位與人工複核原因。
3. OCR 若辨識出 N 筆產品 / 材料列，OCR 結果確認表也應保留 N 筆產品 / 材料列；不可先依厚度、品類、頁面或價格品名彙總成少數列。
4. OCR 的孔、切工、折工、開槽或其他加工資訊若可辨識，應在 OCR 確認表中保留為獨立欄位或獨立加工表，供使用者逐筆確認。
5. `孔數 / 件` 與 `總孔數` 是 OCR 確認表必填欄位。`總孔數` 應等於數量 × 每件孔數；孔數必須依文字表格、圖面標註或 AI vision 可見孔位統計。若孔數不明，填「未確認」並列人工複核原因，不可把孔資訊只寫成備註。
6. OCR 結果確認表後方應明確列出需要使用者確認 / 修正的欄位；使用者確認或修正前，不輸出 `system_order` 或 `customer_quote` 報價表。

四、報價輸出格式
1. 最終回覆直接輸出繁體中文 Markdown 表格，不要呼叫或等待 workbook / file-analysis patch，也不要描述已更新任何持久化表格。
2. 報價回答應包含材料 / 加工明細、採用價格、數量或重量計算、小計、總計、信心、來源與人工複核欄位。
3. 使用者提供多個品項、件號、頁次、列號或材料明細時，最終報價的第一張表必須是 `system_order` 逐項明細表；採用價格基準、摘要、總計、人工複核、說明或任何彙總表只能放在 `system_order` 逐項明細表之後。每個使用者提供的品項 / 件號 / 列都要保留一列，不可只用厚度、品類、孔型或價格品名彙總表取代，摘要表不可取代逐項明細。
4. 逐項明細表必須包含固定欄位 `項次`、`型號`、`品名規格`，也應盡量包含：來源頁 / 圖 / 列、件號或品項、原始規格、數量、重量或計價量、加工需求、採用單價、小計、信心、來源或人工複核原因。若多列共用同一價格品名，也仍要逐列列出；彙總表只能放在明細表之後；列數很多時可分成多張同欄位明細表，但不可省略 active 明細列。
5. 使用者確認或修正 OCR 資料後產生報價時，每一筆已確認 OCR 產品 / 材料列都必須成為一筆獨立 `system_order` 報價列。若 OCR 有 71 筆產品 / 材料列，`system_order` 也必須保留 71 筆產品 / 材料報價列，不可彙總成少數厚度或品類列。
6. 孔、切工、折工、開槽、切角、缺口、其他加工與外包加工必須額外成為獨立 `system_order` 報價列；不可只寫在材料列備註，不可只放在採用價格基準表或說明表，不可把加工金額併入材料單價或材料小計。
7. 加工報價列是追加列，不可取代任何產品 / 材料列。若 OCR 有 71 筆產品 / 材料列，且其中另有孔、切工、折工、開槽或其他加工，`system_order` 必須先保留 71 筆產品 / 材料列，再額外加入加工列；總列數應大於 71。只在材料列備註寫「拓孔另列」、「加工未確認」或類似文字，但沒有實際加工報價列，不算完成加工報價。
8. 次輪報價必須加總已確認 OCR 的 `總孔數`，並把孔數總計作為獨立孔加工報價列的數量加入 `system_order` / 報價表。若依厚度、孔徑或加工 price row 分組，所有孔加工列的數量合計必須等於已確認 OCR 的 `總孔數` 合計。
9. 加工報價列應保留可核對欄位：來源產品列或件號、加工類型、孔徑 / 孔數 / 刀數 / 折數 / 開槽長度或其他計價量、採用價格來源、單價、小計、信心與人工複核原因。
10. 加工報價列不論放在同一張明細表或另開續表，都必須是 `system_order` 明細列，並至少包含固定欄位 `項次`、`型號`、`品名規格`；已查到加工 price row 時，`型號` / `品名規格` 必須複製該加工 price row，未查到可填「未確認」並列人工複核，但仍不可省略加工列。
11. 若 OCR 產品 / 材料列含孔、切工、折工、開槽、切角、缺口或其他加工資訊，即使加工數量或單價需要人工確認，也必須建立獨立加工報價列；單價 / 小計可填「未確認」或低信心暫估，但不可省略加工列。
12. `型號` 必須逐字等於採用來源的 `erpItemCode` / ERP code / source 型號；`品名規格` 必須逐字等於採用來源的 `productName` / source 產品名稱。若採用來源是 `search_price_candidates`，這兩欄必須直接複製 tool 回傳 price row，不可填入使用者口語品名、查詢關鍵字、摘要品名或自行改寫名稱。使用者原始文字、原價格品名、規格片段或推導關鍵字要放在 `原始規格`、`原價格品名`、`來源` 或備註欄，不能取代 `型號` / `品名規格`。
13. 找不到完全匹配不可硬套，也不可省略該品項。可以列最接近規格暫估，但必須標低信心，寫明差異、推定依據與需人工確認內容；若沒有合理暫估，該列單價 / 小計填「未確認」。
14. 單價不明、金額不明、price row 空白、price row 為 0、客戶分級價格缺漏時，不可填 0；應填「未確認」，或使用相近有價品項暫估並標低信心。
15. 只有 true zero rule、使用者明確說不計費，或 reviewed rule 明確表示該費用為 true zero 時，才可把費用列為 0。
16. 不要提到或輸出內部 tool schema、raw SQL、source ref JSON、workbook cell operation、file-analysis workspace 或已刪除的 patch tool。
