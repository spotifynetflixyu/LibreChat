# Steel DB-Backed Quote Rules Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Move Steel task-scoped quote instructions from code into Supabase and add one merged rule lookup tool that returns instruction packets plus quote defaults for one batched interpreted order context.

**Architecture:** `steel.instruction_packets` stores reviewed Admin-editable instruction packets. `lookup_quote_rules` reads `steel.instruction_packets` and `steel.quote_defaults` in one tool call, while `search_price_candidates`, `lookup_formula`, and customer lookup stay separate reviewed data tools. Legacy `lookup_instructions` and `lookup_defaults` remain as compatibility wrappers during migration.

**Tech Stack:** TypeScript in `packages/api`, PostgreSQL/Supabase cloud through `STEEL_POSTGRES_URL`, Zod tool schemas, Jest focused tests.

---

## Runtime Contract

`lookup_quote_rules` is the preferred first rule lookup after AI has an initial interpreted order context. It supports multiple catalog/material keys in one call, for example C 型鋼 plus H 型鋼 plus hole/slotting/cutting facets. The tool returns bounded reviewed packets and defaults; it does not return price rows or decide final prices.

Input shape mirrors the existing batched `lookup_instructions` / `lookup_defaults` context:

- `taskTypes`
- `packetGroupHints`
- `evidenceSummary`
- `catalogContexts`
- `customerContext`
- `reviewState`
- `includeInactive`
- `limit`

Output groups:

- `instructionPacketGroups`
- `instructionPackets`
- `quoteDefaults`
- `requiredLookups`
- `userVisibleNotes`
- `confirmationQuestions`
- `conflicts`
- `sourceRefs`

The AI may include `userVisibleNotes` and `confirmationQuestions` in its answer so the user can see assumptions such as default material, cutting/no-charge rules, non-standard length behavior, or processing details that still need confirmation.

## Database Contract

Create `steel.instruction_packets` in the private `steel` schema. Do not expose it to Supabase Data API clients. Backend and future Admin routes read/write through LibreChat APIs.

Core columns:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `slug TEXT NOT NULL`
- `version INTEGER NOT NULL`
- `title TEXT NOT NULL`
- `locale TEXT NOT NULL DEFAULT 'zh-TW'`
- `packet_groups TEXT[] NOT NULL DEFAULT '{}'`
- `selectors JSONB NOT NULL DEFAULT '{}'`
- `instruction TEXT NOT NULL`
- `blocking_rules TEXT[] NOT NULL DEFAULT '{}'`
- `required_lookups TEXT[] NOT NULL DEFAULT '{}'`
- `user_visible_notes TEXT[] NOT NULL DEFAULT '{}'`
- `confirmation_questions TEXT[] NOT NULL DEFAULT '{}'`
- `priority INTEGER NOT NULL DEFAULT 100`
- `confidence TEXT NOT NULL DEFAULT 'medium'`
- `active BOOLEAN NOT NULL DEFAULT true`
- `review_state TEXT NOT NULL DEFAULT 'reviewed'`
- `source_refs JSONB NOT NULL DEFAULT '[]'`
- lifecycle fields: `supersedes_packet_id`, `published_at`, `invalidated_at`, `created_by`, `reviewed_by`, `reviewed_at`, `created_at`, `updated_at`

Constraints:

- unique active/reviewed packet identity through `(slug, version)`
- `confidence IN ('low', 'medium', 'high')`
- `review_state IN ('draft', 'needs_review', 'reviewed', 'rejected')`
- JSON type checks for `selectors` and `source_refs`
- invalidated rows must be inactive
- no self-supersede

Indexes:

- `(review_state, active)`
- GIN on `packet_groups`
- GIN on `selectors`
- `(priority, id)`
- `(slug, version)`

## Admin Boundary

The migration only creates backend storage and runtime retrieval. Admin UI can be built later on top of the table. When Admin UI arrives, it should:

- edit drafts without changing active reviewed rows directly
- review/publish a new version
- deactivate or supersede old versions
- keep `source_refs` pointing to import files, docs, or Admin review notes
- never require code edits for C 型鋼/H 型鋼/角鐵 prompt rules

## Compatibility

Keep existing tools callable:

- `lookup_instructions`: reads `steel.instruction_packets` and returns only instruction-related output.
- `lookup_defaults`: reads `steel.quote_defaults` and returns only default-related output.
- `lookup_quote_rules`: calls both repositories and returns merged output.

Provider policy should prefer `lookup_quote_rules` before category-dependent lookup. Existing live smoke names can be updated later; old tests can still assert instruction packets are retrieved before price lookup.
